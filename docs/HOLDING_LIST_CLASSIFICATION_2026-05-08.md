## Classification scheme

Four lists, each with the same column shape (`ID | Description | Surface | Notes | Server/AAB`):

- **Bugs (B)** — broken or incomplete behavior on a user-facing surface
- **Features (F)** — new user-facing capabilities
- **Tooling (T)** — internal dev / test / measurement infrastructure
- **Ideas (I)** — brainstorming-stage entries; deferred-by-design or path-not-chosen items not yet committed as real features

**Severity** — encoded in the ID: 1 = top, 2 = medium, 3 = low. Letter suffix (`a`, `b`, `c`…) disambiguates within a severity tier (e.g., B1a, B1b, B1c are all top-severity bugs).

**Server/AAB** — where the work lands:
- `Server` — Edge Function, SQL, voice server, web; no AAB build required
- `AAB` — mobile code; requires `npx eas build --auto-submit`
- `Both` — server + mobile pieces

**Surface** — which user-facing surface owns the work; used for cross-surface drift discipline (see CLAUDE.md Rule 16). Values:
- `mobile` — work lands on mobile codebase only; no voice change required
- `voice` — work lands on voice-server codebase only; no mobile change required
- `both` — both surfaces; parity required (when one ships, the other must follow before drift)
- `backend` — shared backend (Edge Functions / SQL); both surfaces benefit automatically
- `website` — `mynaavi-website` repo only; neither mobile nor voice

**Architectural principle (Wael 2026-05-08):** every queryable channel = background sync at per-channel depth + live-overlay at question-time. Applies to calendar, email, SMS, WhatsApp.

**Cross-surface drift mechanism:** classification's `Surface` tag is best-effort interim discipline. The mechanical guarantee comes from Voice Completion Roadmap W2 (Anthropic Structured Outputs) + W3 (Voice Automated Regression Suite) — see `docs/VOICE_COMPLETION_ROADMAP_2026-05-08.docx`. Until W2 + W3 land, Surface column + CLAUDE.md Rule 16 (`parity-impact:` on commits) are the human-discipline net.

## Shipped this session (2026-05-26)

- **Google Play Health apps declaration — RESOLVED.** Google's automated AAB scanner flagged Naavi's manifest as containing `android.permission.ACTIVITY_RECOGNITION` (contributed transitively by the Transistorsoft `react-native-background-geolocation` library, which uses Android's Motion API for battery-efficient background geofencing). Google's own form copy directs non-health apps to *"remove this permission from your app's manifest."* Investigation took two paths in this session:

  - **Path B — strip the permission via Expo config plugin (TESTED, FAILED).** Built preview APK V57.22.4 build 200 with new `plugins/withRemoveActivityRecognition.js` (Android manifest merger override using `tools:node="remove"` to drop ACTIVITY_RECOGNITION and FOREGROUND_SERVICE_HEALTH from the merged manifest) + paired `disableMotionActivityUpdates: true` in `BackgroundGeolocation.ready()` so the SDK would stop calling the now-blocked Motion API. Motion API is not a battery optimization — it is the technical signal MyNaavi uses to wake from stationary mode and poll GPS for geofence evaluation. Reverted in commit `2c7f803` (V57.22.5 build 201). The `withRemoveActivityRecognition.js` plugin is deleted; SDK config restored.

  - **Path A — keep permission, complete the Health apps declaration form (SUBMITTED, ACCEPTED).** Wael submitted a 6-section justification on Google Play Console: (1) what MyNaavi is (mobile assistant accepting natural-language requests); (2) features available today — Gmail integration, Google Calendar integration, Google People API contacts, Google Drive document storage, time- and location-based reminders, lists, memory, voice phone access, morning brief — each mapped to a connected Google service a reviewer can verify; (3) why ACTIVITY_RECOGNITION is needed — Motion API signal to wake from stationary mode for geofence evaluation, with empirical confirmation of non-functionality without it (citing the Path B drive-test result); (4) explicit denial of motion-data collection / display / sharing / persistence; (5) mapping to Google's six approved health-permission use cases (per https://support.google.com/googleplay/android-developer/answer/12991134) with each explicitly denied as not applicable to MyNaavi; (6) reviewer-verification steps (chat interface, Alerts screen, Settings screen, Drive folder structure). Plus a 250-char "Other" justification in the form's short field. **Google accepted the declaration.** The Health-form blocker on the App Content page is cleared. Internal Testing track unblocked for future submissions.

- **ACTIVITY_RECOGNITION decision permanently closed.** Two paths tested in this session; Path B is dead per drive evidence (geofencing non-functional without Motion API on Samsung at 300m radius). The permission is required for the location-reminders feature, regardless of battery cost. The Path A justification is on file with Google and accepted. Net: no code changes remain at session close; ACTIVITY_RECOGNITION stays in the manifest.
