## 1. What exactly is broken

`action_rules_user_label_unique` — a UNIQUE constraint on `(user_id, label)`, live in production — blocks creating any new action rule whose generated `label` text exactly matches a **prior** rule's label for that user, **regardless of whether the prior rule is still enabled**. Since Claude generates the same literal label text for the same phrasing (e.g., "SMS to 343-333-2567 in 3 minutes"), any user who repeats the same alert request on two different occasions — even long after the first alert already fired, delivered, and was disabled — gets permanently blocked from creating the second one, with a 409 that (independent of this finding — see B9y/Track B-1e) may not even be surfaced to them truthfully.

This is separate from two related-but-distinct issues already tracked: the untracked-migration finding itself (logged to [[project_naavi_architecture_integrity_audit]] T1a — "this constraint isn't in git" is T1a's concern) and the silent-failure-on-write defect (Track B-1e, already fixed — "a failed write wasn't reported truthfully" is that ticket's concern). B9z is specifically: **should a repeat of the same alert request permanently fail at all, and if not, what's the correct fix.**

## 2. Evidence

**The constraint's exact definition** (Supabase SQL editor, `pg_indexes`, 2026-07-16):
```sql
SELECT (indexdef ~* 'WHERE') AS has_where_clause,
       substring(indexdef FROM 'WHERE (.*)$') AS where_condition
FROM pg_indexes WHERE indexname = 'action_rules_user_label_unique';
```
Result: `has_where_clause = true`, `where_condition = ((label IS NOT NULL) AND (label <> ''::text))`.

**What this proves:** the `WHERE` clause exists only to allow multiple NULL/empty-label rows to coexist (a standard partial-index idiom) — it has **no `enabled = true` scoping**. The constraint is enforced across every row with a non-empty label, active or not.

**Direct reproduction, same day:** a live voice test ("text me at [phone] in 3 minutes") failed with:
```
ERROR: duplicate key value violates unique constraint "action_rules_user_label_unique"
Key (user_id, label)=(788fe85c-b6be-4506-87e8-a8736ec8e1d1, SMS to 343-333-...
```
against `bcb5e02d-7bcc-44fe-a63e-819a625911ee` — a row created the **previous day**, `enabled = false` (already fired or otherwise deactivated), label `"SMS to 343-333-2567 in 3 minutes"` — identical text to the new request.

**Observed deterministic behavior for identical tested phrasing** (narrowed per Phase 3 review — the evidence supports this exact claim, not a broader one): two independent reproductions of the same spoken request generated byte-identical labels a full day apart, in separate calls, separate conversations, demonstrating that repeated requests of this form collide under the current constraint. This is not yet proven to generalize to label generation *in general* — only to this tested phrasing shape. The label-building logic itself (voice: `naavi-voice-server/src/index.js`'s action construction; not yet traced to the exact function/line — flagged for Phase 2) has not been read directly to confirm the mechanism, only inferred from the two matching outputs.

**Scope beyond voice, not yet verified:** `manage-rules` (mobile-side) also writes to `action_rules` and is independently confirmed to be subject to this exact constraint — `docs/SESSION_HANDOFF_2026-06-14_V115_AAB_NEXT.md` documents a *different* bug from 2026-06-14 where this same constraint fired because `manage-rules`' disambiguation branch left `label` unset, falling back to a generic `'Action rule'` string that collided across unrelated rules. That's a different trigger (missing label vs. identical label) hitting the same underlying constraint — corroborates this is a general `action_rules` write-path issue, not voice-specific, but the mobile-side identical-label reproduction itself hasn't been directly tested this session.

## 4. What alternatives were considered

- **"Maybe this is intentional — a genuine duplicate-alert guard, working as designed."** Not ruled out as a *possible* original intent, but the guard (if intentional) doesn't distinguish "this exact alert is still active" (a real duplicate worth blocking) from "this exact alert already fired and is long gone" (not a duplicate in any meaningful sense) — the lack of `enabled` scoping means it can't tell the two apart. Even under the most charitable reading of original intent, the current behavior over-blocks.
- **"Maybe the label text isn't actually meant to be the dedup key, and this constraint is a leftover from before the per-trigger-type partial indexes (`20260525000000_compliance_fixes.sql`) were added."** Plausible given `action_rules_user_label_unique` is *not* in that migration or any other tracked one, but not confirmed — would need someone with deploy history knowledge (Wael, or Supabase's own point-in-time schema history if available) to confirm when/why it was added.
- **"Maybe this only affects voice, not mobile."** Ruled out by the `manage-rules` handoff evidence (§2, above) — mobile hits the same constraint, just via a different trigger (missing label, not identical label).

## 6. Next step

Phase 2 — Change Planning, per governance. Candidate directions, not yet designed or chosen:
1. **Add `enabled = true` scoping to the constraint** — mirrors the pattern already used by the per-trigger-type partial indexes in `20260525000000_compliance_fixes.sql`. Would need a migration (`DROP` + re-`CREATE` with the added `WHERE`), and — since the current constraint isn't in git at all — this migration would also need to formally adopt it into version control for the first time, closing the T1a instance in the same change.
2. **Stop using label text as a dedup key entirely** — redesign toward a logical key that doesn't depend on freeform generated text (e.g., something closer to the `(user_id, trigger_config->>'datetime')` pattern already used elsewhere, or a proper content-hash/fingerprint of the semantic request). Larger change, touches more of the write path.
3. **Accept current behavior, do nothing** — if investigation shows this was genuinely intentional and the "permanent block" framing is judged acceptable risk (e.g., if it's rare enough in practice). Needs an explicit decision, not a default.

**Design principle Phase 2 should evaluate all three candidates against (added per Phase 3 review):** a uniqueness constraint should enforce the actual business rule, not an implementation artifact of how that rule happened to get built. The business rule here appears to be *"don't create duplicate active alerts."* The implementation as it stands enforces a different rule — *"don't reuse identical generated label text"* — and those are not the same thing: two semantically different alerts could coincidentally generate the same label and get wrongly blocked, while the same alert reworded slightly would generate a different label and bypass dedup entirely, in either direction. Phase 2 should choose among the three candidates by which one closes that gap between implementation and intent, not by which migration is easiest to write.

Phase 2 should also confirm scope 1's migration-drift angle with T1a before designing, since the two tickets' fixes may need to land together (adopting the constraint into git and fixing its scoping in the same change, rather than two separate migrations touching the same object).
