Purpose

This document defines the normalized internal data model for Naavi. The goal is to establish provider-agnostic data interfaces that sit between the Naavi intelligence layer and any external service provider (Google, Microsoft, Apple, Dropbox, etc.).

Why This Matters

Today Naavi's integration layer is tightly coupled to Google services. Every function call in the codebase references a specific provider. This creates a structural problem: adding a second client who uses Outlook, iCloud, or OneDrive requires rewriting large portions of the application.

The normalized model solves this by making Naavi own the data concepts. Providers become interchangeable adapters underneath.

Architecture Principle

User (Robert / Future Clients) ↓ Naavi Orchestration (AI Layer) ↓ Normalized Internal Data Model — this document ↓ Provider Adapters — Google, Microsoft, Apple, Notion, etc. ↓ External APIs

The UI and AI layers never talk to a provider directly. They only speak Naavi's normalized language. When a new provider is added, only a new adapter file is written — nothing else changes.

Open Decision — Before Implementation Begins

Every adapter call routes through the user's provider preference. The following must be decided before writing any interface:

Should one user be able to have Google Calendar + Outlook Calendar active simultaneously (multi-provider per category)?
Or one active provider per category at a time (single-provider per category)?

OptionField TypeComplexityCovers
Single active providerstringLower95% of use cases
Multiple simultaneousstring[]HigherPower users, migrations
Recommendation: Start with single active provider per category. The field can be migrated to an array later without breaking the interface contract.

Implementation Sequence

Once this document is approved, the recommended build sequence is:

PhaseTaskEffort
Phase 1Write TypeScript interfaces for all 9 entities — no implementation changes to existing code1 day
Phase 2Wrap existing Google libs behind adapter classes implementing those interfaces3–5 days
Phase 3Update UI and orchestration layer to import adapters, not Google libs directly2–3 days
Phase 4Add new provider adapters as new clients require themPer provider
Phase 1 and 2 produce zero visible change to the user. The app behaves identically. The value is entirely in the structural clean-up that makes Phase 4 fast and safe.

Non-API Provider Integration

Not every system Naavi may need to connect to will have a public API. This is common in healthcare, government, enterprise, and legacy environments. The Provider Adapter Pattern specifically supports this — the adapter interface only cares about the output, not how the data was retrieved.

// From Naavi's perspective, both of these look identical:
GoogleCalendarAdapter.fetchEvents() → CalendarEvent[]  // REST API
HospitalCalendarAdapter.fetchEvents() → CalendarEvent[]  // Browser automation
ICSFileAdapter.fetchEvents() → CalendarEvent[]  // File parsing

// The rest of the system never knows the difference.

Integration Methods

📡
Protocol Adapters
No vendor API required
Standard open protocols supported by most enterprise and healthcare systems. The broadest coverage with the least fragility — no login scraping, no brittle selectors.
CalDAV — any calendar CardDAV — any contacts IMAP / SMTP — any email WebDAV — any file storage
📁
File-Based Adapters
Export → watch → import
Many systems can export data to standard file formats. Naavi watches a folder or email inbox and imports automatically when a new file appears.
ICS calendar files CSV contacts PST email archives PDF reports + OCR
🌐
Browser Automation
Puppeteer / Playwright
A headless browser logs into a web application and extracts data programmatically. Works on any web interface regardless of whether a public API exists.
Hospital patient portals Government systems Insurance portals Internal intranets
🖥️
Local Desktop Agent
Small background app on device
A lightweight background application installed on the user's machine reads from local desktop apps and syncs to Naavi via a secure local API. No cloud dependency.
Outlook desktop Apple Mail / Calendar Local EHR software QuickBooks desktop
📨
Webhook / Email Parsing
Parse inbound notifications
Many legacy systems send email or webhook notifications for events. Naavi receives these and parses the structured content — no outbound API call needed at all.
Lab result emails ERP alert emails Booking confirmations System webhooks
🤖
RPA Bridge
Robotic Process Automation
Tools like Microsoft Power Automate or UiPath act as a bridge between Naavi and completely closed enterprise systems. The RPA tool handles the proprietary interaction, Naavi receives normalized data.
Power Automate UiPath Zapier webhooks Make (Integromat)

Key Architectural Principle

The adapter interface is the firewall. Browser automation code, file watchers, RPA bridges, and OCR pipelines all live entirely inside the adapter. They never leak into the UI, the AI orchestration layer, or any other entity. Adding a non-API provider is always one new adapter file — nothing else in the system changes.

05
Adapter Type Values
How each adapter retrieves data — transparent to the rest of the system
ValueMeaningMaintenance RiskWhen to Use
"api"Provider has a public REST or GraphQL API with official credentialsLow — provider manages versioningDefault. Google, Microsoft, Dropbox
"protocol"Uses an open standard protocol (CalDAV, CardDAV, IMAP, SMTP, WebDAV)Very low — protocols are stable for decadesApple Calendar, Apple Contacts, any IMAP email
"file"Reads from exported files (ICS, CSV, PST, PDF) placed in a watched folderLow — file format rarely changesOffline systems, legacy exports, manual data entry
"automation"Headless browser logs into a web UI and extracts dataHigh — breaks when UI changesLast resort for web-only systems with no API or protocol
"agent"Local background app installed on user's device reads from desktop applicationsMedium — OS and app updates may break itOutlook desktop, Apple Mail, local EHR software
"rpa"Microsoft Power Automate, UiPath, or Zapier acts as a bridgeMedium — depends on RPA tool stabilityClosed enterprise systems, government portals
Rule: Always choose the method highest up this list that is available for a given provider. "api" beats "protocol" beats "file" and so on. "automation" should only be used when no other method exists — it is the most fragile.
