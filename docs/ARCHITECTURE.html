1. What Naavi Is

Naavi is a life orchestration companion for active seniors. The primary user is Robert, 68, Ottawa. Robert's problem is that his tools — calendar, Gmail, Drive, health records, contacts, maps — do not talk to each other. Naavi connects them through a single conversational interface: he speaks or types, Naavi responds with voice and executes actions autonomously on his behalf — no app switching, no copy-pasting, no token management.

Zero-friction principle: Robert never authenticates, reconnects, manages tokens, or copies data between tools. Everything runs server-side automatically.

6. Data Flow — One Full Conversation Turn

1. Robert taps microphone

2a. Whisper path:
    useWhisperMemo -> records MediaRecorder clip -> base64 encodes
    -> transcribe-memo Edge Function -> OpenAI Whisper API
    -> transcript text

2b. Web Speech path:
    useVoice (continuous=true) -> SpeechRecognition API (browser-native)
    -> accumulates finalParts[] -> delivers on onend
    -> transcript text

3. useOrchestrator.send(transcript)
   Per-turn card arrays initialised: turnNav[], turnDrive[], turnDrafts[],
   turnEvents[], turnDocs[], turnMemory[]

4. Person context injection (lib/memory.ts):
   extractPersonQuery() — regex with stopwords filter, spaced-letter normalisation
   if person name found -> getPersonContext() queries calendar + Notion + Gmail + Drive
   -> formatPersonContext() appends context block to message

4b. Phone number lookup (lib/contacts.ts):
   Digits extracted from message — if 10-digit number detected:
   -> lookupContactByPhone() tries 4 formats via Google People API
   -> if found: contact name/email/phone injected into message
   -> if not found: "no contact found" injected so Claude answers accurately

5. naavi-client.ts -> sendToNaavi():
   -> callNaaviEdgeFunction()
   -> POST to naavi-chat Edge Function {system, messages, max_tokens:2048}
   -> Edge Function holds ANTHROPIC_API_KEY server-side
   -> calls claude-sonnet-4-6

6. Claude returns raw text

7. parseResponse() - 4-pass parser:
   extractJsonBlocks(): finds all top-level {...} blocks, returns last-first
     (handles Claude self-corrections — most recent block is tried first)
   Pass 1: JSON.parse each block in order
   Pass 2: fix literal newlines inside string values
   Pass 3: strip control characters, fix trailing commas, quote unquoted keys
   Fallback: regex-extract speech value as last resort
   fixSentLanguage(): corrects "I sent" -> "Draft is ready" when DRAFT_MESSAGE present

8. useOrchestrator executes actions:
   SAVE_TO_DRIVE       -> save-to-drive Edge Function + saveDriveNote() to naavi_notes
   REMEMBER            -> ingestNote() -> ingest-note Edge Function
                          -> Claude extracts fragments -> OpenAI embeds -> knowledge_fragments
   CREATE_EVENT        -> create-calendar-event Edge Function -> Google Calendar
   FETCH_TRAVEL_TIME   -> get-travel-time Edge Function -> Google Maps Distance Matrix
                          -> turnNav[] card with destination + durationMinutes + leaveByMs
   DRIVE_SEARCH        -> search-google-drive Edge Function -> turnDrive[] file cards
   DRAFT_MESSAGE       -> accumulated in turnDrafts[]
   ADD_CONTACT         -> saveContact() + savePerson() -> Supabase
   SET_REMINDER        -> saveReminder() -> Supabase
   LOG_CONCERN         -> saveTopic() -> Supabase
   UPDATE_PROFILE      -> saveTopic() -> Supabase
   SCHEDULE_MEDICATION -> calculate all active dose dates from on_days/off_days/duration
                          -> CREATE_EVENT for each dose (batched sequentially)
                          -> 💊 emoji events on Google Calendar

9. setTurns() appends ConversationTurn with all card arrays for this turn

10. speakResponse() - voice output
    Web: Web Speech Synthesis
         getVoices() — waits for onvoiceschanged if voices not yet loaded
         Priority: Microsoft Aria Online (Natural) > Google UK English Female
                   > Google US English > Samantha (Safari) > any en-CA
         Rate: 0.88
    Native: expo-speech. Rate: 0.85

11. index.tsx renders turns[]:
    For each turn: user bubble -> assistant bubble -> action cards
    Cards are scoped to their turn — no bleed between conversations

7. Morning Brief Pipeline

App mount
  -> fetchOttawaWeather() (immediate, no auth)
  -> supabase.auth.getSession() -> currentUserId
  -> Promise.all([
       fetchUpcomingEvents(7, userId),   -- calendar_events table
       fetchUpcomingBirthdays(userId),   -- calendar_events (birthday type)
       fetchImportantEmails(userId),     -- gmail_messages (is_important = true)
     ])
  -> enrichWithTravelTime(calendarItems)
       For each event with location + startISO within 8 hours:
       -> get-travel-time Edge Function
       -> Google Maps Distance Matrix API (server-side)
       -> returns { durationMinutes, distanceKm, leaveByMs, summary }
  -> setBrief() -- grouped, collapsible categories

Background sync (parallel):
  -> sync-google-calendar + sync-gmail Edge Functions
  -> re-fetch + re-enrich -> setBrief() updated silently

Leave-time alert timer (setInterval 30s):
  now >= leaveByMs AND now <= startMs + 60min:
  -> green banner "Time to leave" or red banner "Running late"
  -> tap -> opens Google Maps navigation

9. Knowledge & Memory Layer

A semantic long-term memory system combining Claude for classification and OpenAI for vector embeddings stored in pgvector.

Write Path (REMEMBER action)

Robert: "Remember that I prefer aisle seats on flights"
  -> Naavi returns REMEMBER action: { "text": "Robert prefers aisle seats on flights" }
  -> ingestNote(text, 'stated') -> ingest-note Edge Function
     1. Claude claude-sonnet-4-6 with classifier prompt
        -> extracts: [{ type, content, classification, confidence }]
        Types: life_story | important_date | preference | relationship | place | routine | concern
        Classifications: PUBLIC | PERSONAL | SENSITIVE | MEDICAL | FINANCIAL
     2. OpenAI text-embedding-3-small (1536 dimensions) per fragment
     3. INSERT into knowledge_fragments with user_id + embedding vector
  -> UI: purple card "Saved to memory -- N fragments stored"

Read Path (Person Context Injection)

Robert: "Tell me about my meeting with Sarah"
  -> extractPersonQuery() -- regex with stopwords + spaced-letter normalisation
  -> getPersonContext() queries calendar + Notion + Gmail + Drive
  -> formatPersonContext() injects into message:
     "## What Naavi knows about Sarah
      - Relationship: colleague
      - Email: sarah@example.com
      - Upcoming meetings: (1)
      - Past meetings: (3 in last 6 months)
      - Recent emails: (2)"
  -> enrichedMessage sent to Claude

10. Contact Resolution (4-Tier)

Robert: "Send an email to Heaggan"

By Name:
Tier 1: SELECT name, email FROM contacts
         WHERE user_id = ? AND name ILIKE '%heaggan%'
  -> (not found)

Tier 2: SELECT from_email FROM gmail_messages
         WHERE user_id = ? AND from_name ILIKE '%heaggan%' LIMIT 1
  -> (not found)

Tier 3: Google People API -- people.searchContacts?query=heaggan
         Requires: contacts.readonly scope + People API enabled in GCP
  -> found: { name, email, phone }

By Phone Number:
When a 10-digit number is detected in Robert's message:
  -> lookupContactByPhone() tries 4 formats via Google People API:
     "613-769-7957", "6137697957", "+16137697957", "+1 613 769 7957"
  -> searchContacts queries across names, emails, AND phone numbers
  -> result injected into Claude's context before generating response

Spaced-letter input:
  "find contact name w a e l" -> normalised to "wael" before pattern matching

11. Medication Scheduling

Robert: "The doctor told me to take Metformin twice a day,
         5 days on 3 days off, starting tomorrow for one month"

1. Claude emits SCHEDULE_MEDICATION action:
   { type, name, dose_instruction, times: ["08:00","20:00"],
     on_days: 5, off_days: 3, start_date, duration_days: 30 }

2. Orchestrator calculates all active dose dates:
   Cycle: days 1-5 active, days 6-8 pause, days 9-13 active, ...
   30 days / 8-day cycle = ~19 active days x 2 doses = ~38 events

3. Creates individual Google Calendar events:
   Title: "💊 Metformin"
   Duration: 30 min blocks at each dose time
   Batched sequentially to avoid rate limits

4. All events visible in Google Calendar immediately
Phase B (planned): medications Supabase table + daily cron + push notifications. Option A (current) creates individual calendar events. Simple and works today — upgrade path is clear when push notifications are added.

13. Security Model

API Keys — Server-Side Only

  • ANTHROPIC_API_KEY — Supabase Edge Function secret
  • OPENAI_API_KEY — Supabase Edge Function secret
  • GOOGLE_MAPS_API_KEY — Supabase Edge Function secret
  • Device holds only: Supabase anon key (public by design) + Supabase URL

Authentication

  • Google OAuth via Supabase Auth
  • Google refresh token captured at SIGNED_IN event → stored in google_tokens table server-side
  • App never holds the refresh token after sign-in

JWT / RLS

  • All Edge Functions: verify_jwt = false in config.toml
  • Auth handled inside each function via userClient.auth.getUser()
  • All tables: RLS enabled, policies enforce auth.uid() = user_id
Known issue resolved: JWT key rotation (HS256 → ECC) caused 401s at the Supabase gateway. Fix: disable gateway-level JWT verification and handle auth inside each Edge Function via RLS. This is now the standard pattern for all functions.

17. Recommendations & Roadmap

Immediate (Next Sprint)

1. Automatic knowledge retrieval at conversation start
The search-knowledge Edge Function exists but is not wired into the morning brief. Recommended: at app load, run searchKnowledge("today morning context") and inject the top 5 relevant fragments into the system prompt. Naavi would then recall Robert's preferences automatically without being asked.

2. Confirmation cards for SET_REMINDER, LOG_CONCERN, UPDATE_PROFILE
These actions are stored but show no visual feedback. Robert has no confirmation that his reminder or concern was saved. Low-effort, high-trust improvement.

3. Medication scheduling — Phase B upgrade
Current implementation (Option A) creates individual calendar events. Phase B: medications Supabase table + daily cron + push notifications. Enables cancel/modify ("stop taking Metformin") and reminders that work when the app is closed.

Medium-Term

4. Push notifications — Web Push API + Supabase pg_cron triggered at leaveByMs and medication dose times. Currently the leave-time banner and medication reminders only work when the app is open.

5. Native mobile app (EAS Build) — Expo EAS Build produces an installable .ipa / .apk with no code changes. Gives Robert a home-screen icon and background push notifications on iOS.

6. Single voice button with automatic fallback — Two buttons cause UX confusion. Consolidate: try Web Speech API first, fall back to Whisper if unsupported or silent for 3 seconds.

7. Health integrations — Withings (scale + blood pressure) REST API. MyChart FHIR R4. Would make the health brief category genuinely useful.

8. Twilio SMS — Add DRAFT_SMS action type for contacts who don't use email.

Architectural

9. Conversation persistence across sessions — The history state resets on tab close. Persist conversation turns in Supabase; load last N turns at app start. Enables "as we discussed yesterday…" continuity.

10. Multi-user / caregiver mode — Extend Supabase RLS with a caregivers table and shared read policies. No data model restructuring required. A family member or home care worker could have a read-only view of Robert's brief and concerns.
